Architectural Perspectives

We have come up with two potential architectures for integrating WPS capabilities with Apache Airavata. These perspectives address the limitations of existing systems which are identified through out literature review. The 1st architecture is focused on enabling Apache Airavata to orchestrate WPS processes by importing WSDLs of an existing WPS instances. The 2nd architecture is about deploying a workflow created in Apache Airavata as a WPS which provides a high degree ofreusability for geoscience experiments.

WPS Orchestration

WPS orchestration is about providing the flexibility to the chain a set of WPS processes. It is a crucial need for geoscientist to carry out their experiments effectively. Hence, WPS orchestration is a major functionality that should be present in a geoscience gateway. Figure 1 shows the architectural diagram of WPS orchestration using Apache Airavata with the use of SOAP interfaces provided by the existing WPS implementing software. PyWPS WPS implementation is capable of expose WPS services through WSDL which can use to import WPS services as components in XBaya UI thourgh WSDL interface to chain WPS processes. 52 o North WPS implementation is still lacking feature of expose WPS services through WSDLs. But it‟s in the development roadmap of 52o North WPS implementation.

Functional flow

  • Apache Airavata is capable of importing a WSDL, listing available services of a WPS          instance (PyWPS, 52o North WPS), through its XBaya GUI.
  • Each service imported through the WSDL is made available as a component which can be used to compose a workflow simply through drag-and-drop in XBaya UI.
  • After creating the workflow, Apache Airavata‟s workflow interpreter engine can execute it through SOAP messages consuming the used WPS processes.

Objectives

  • Reusing existing WPS processes in a geoscience application
  • Ease of WPS orchestration through scientific workflows
  • Use existing WPS processes available in Desktop GIS systems like GRASS from theweb by bridging with WPS implementations

Workflows in WPS Standard

Geoscience experiments are often complex and requires a significant programming knowledge to develop sophisticated algorithms and programs for them. Therefore at the current stage of computational geoscience, geoscientists are unable to engage in such tasks on their own without the aid of a computer scientist. But it would be convenient and beneficial for the geoscientists to be able to author a geoscience process on their own.

Complex geoscience computations require HPC resources for executing. General purpose computers are not sufficient to carry out these complex computations. Apache Airavata enables the support to execute the complex scientific applications deployed in grid environments and supercomputing resources. Apache Airavata‟s GFac component wraps command line driven scientific applications and make them into network-accessible services which can be used to create scientific workflows. A geoscience gateway is a place which abstracts the complex computational resources and provides a domain specific environment for scientist to conduct their experiments. Therefore architecture 2 addresses a major functionality which needs to be present in a geoscience gateway.

Functional flow

  • Geoscientist author and test workflows in XBaya UI, making use of the services added by the gateway developer
  • Geoscientist deploy the created workflows in the WPS server instance (PyWPS)
  • Deployed WPS processes can be queried by any WPS client by sending a <GetCapabilities> request to the server instance. This request will be responded directly by the WPS server itself without communicating with Apache Airavata
  • Queries to WPS for process description (<DescribeProcess>) will also be responded directly by the WPS server.
  • Execute requests to WPS server (<Execute>) will be forwarded to Apache Airavata, which will only handle the computation. The process output will be then sent to Apache Airavata and forwarded to WPS server accordingly.These steps are explained in Figure 2.

Objectives

  • Ability to share computations created as workflows by deploying them in a WPS server
  • Use the functionalities provided in various geo-processing libraries (GDAL/OGR,Proj4) available via WPS implementations.
  • Authoring of processes in WPS standard which is now in the hands of the software developer can be made
